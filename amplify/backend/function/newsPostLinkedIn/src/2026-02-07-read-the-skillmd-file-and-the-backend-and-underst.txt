⏺ Yes, it's technically possible, but there are important constraints to
  understand first.

  LinkedIn Posting Options

  Option A: LinkedIn API (Official)

  How it works:
  1. Create a LinkedIn Developer App at https://www.linkedin.com/developers/
  2. Get OAuth 2.0 tokens (requires user login consent)
  3. Use the Posts API to publish content programmatically

  Requirements:
  - A LinkedIn Developer App with the w_member_social permission (personal
  profile) or w_organization_social (company page)
  - OAuth 2.0 access token (expires, needs refresh)
  - LinkedIn must approve your app for the required scopes

  Limitation: LinkedIn has significantly restricted API access. The
  w_member_social scope is only available to apps that have been
  verified/approved through their developer program, which can take time.

  Option B: Add a Lambda that prepares the post + uses the API

  You could add a 4th Lambda to the pipeline that:
  1. Runs after NewsProjectInvokeAgentLambda completes
  2. Reads the freshly generated topics + summaries from DynamoDB
  3. Formats a LinkedIn-friendly post (e.g., top 3 topics with brief summaries)
  4. Posts via LinkedIn's REST API using a stored OAuth token

  Option C: Semi-automated (most practical)

  Generate the LinkedIn post content automatically, but copy/paste manually or
  use a scheduling tool like Buffer/Hootsuite. This avoids LinkedIn's API
  approval process entirely.

     Plan: Auto-Post New Topics to LinkedIn

     Context

     The Global Perspectives pipeline generates ~13 AI-analyzed news topics
     hourly. The user wants new topics automatically posted to their
     personal LinkedIn profile with the summary and chain reaction
     prediction. Topics that have already been posted should be skipped.

     Architecture

     A new 4th Lambda (newsPostLinkedIn) runs after the existing pipeline
     completes:

     EventBridge (hourly)
       -> newsInvokeGemini         (:00 - fetch news, cluster topics)
       -> NewsProjectInvokeAgentLambda  (:05 - generate AI content, swap
     staging->latest)
       -> newsPostLinkedIn         (:15 - detect new topics, post to
     LinkedIn)  <-- NEW

     Prerequisites (Manual, Before Implementation)

     1. Create LinkedIn Developer App

     1. Go to https://www.linkedin.com/developers/apps -> Create App
     2. Link it to a LinkedIn Page you admin
     3. Under "Products" tab, request "Share on LinkedIn" and "Sign In with
     LinkedIn using OpenID Connect"
     4. Note the Client ID and Client Secret
     5. Add redirect URL: http://localhost:3000/callback

     2. One-Time OAuth Authorization

     1. Open in browser (replace YOUR_CLIENT_ID):
     https://www.linkedin.com/oauth/v2/authorization?response_type=code&clie
     nt_id=YOUR_CLIENT_ID&redirect_uri=http://localhost:3000/callback&scope=
     openid%20profile%20w_member_social
     2. Authorize -> copy the code from the redirect URL
     3. Exchange code for tokens:
     curl -X POST https://www.linkedin.com/oauth/v2/accessToken \
       -d 'grant_type=authorization_code&code=AUTH_CODE&client_id=ID&client_
     secret=SECRET&redirect_uri=http://localhost:3000/callback'
     4. Get person ID:
     curl -H "Authorization: Bearer ACCESS_TOKEN"
     https://api.linkedin.com/v2/userinfo
     5. Store tokens in AWS SSM Parameter Store:
     aws ssm put-parameter --name "/globalPerspectives/linkedin/accessToken"
      --type SecureString --value "TOKEN"
     aws ssm put-parameter --name
     "/globalPerspectives/linkedin/refreshToken" --type SecureString --value
      "TOKEN"
     aws ssm put-parameter --name "/globalPerspectives/linkedin/personId"
     --type String --value "PERSON_ID"
     aws ssm put-parameter --name
     "/globalPerspectives/linkedin/tokenExpiresAt" --type String --value
     "2026-04-07T00:00:00Z"

     3. Create DynamoDB Table for Post Tracking

     aws dynamodb create-table \
       --table-name NewsProject-linkedin-posts \
       --attribute-definitions AttributeName=PK,AttributeType=S \
       --key-schema AttributeName=PK,KeyType=HASH \
       --billing-mode PAY_PER_REQUEST \
       --time-to-live-specification Enabled=true,AttributeName=ttl

❯ do i need a ttl for the post table in dynamoDB? 

⏺ Yes, I'd recommend it. The TTL is already built into the Lambda code - it sets
   a 30-day TTL on each record (line 251 of index.js).

  Without TTL enabled on the table, those records just pile up forever. With it
  enabled, DynamoDB auto-deletes records after 30 days, so if the same topic
  resurfaces a month later with new developments, it can be posted again.

  In your DynamoDB table settings:
  - TTL attribute name: ttl
  - Enable it under the table's Additional settings > Time to Live
